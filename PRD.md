# Draw Dog PRD

## 1. 產品概述

**產品名稱（暫定）**：Draw Dog

**產品定位**：
一款受「Save the Dog / Save My Pets」類型啟發的輕量網頁益智遊戲。玩家用手指或滑鼠畫出一筆線條，形成保護屏障，阻止蜜蜂攻擊狗狗；只要在限定時間內成功保護狗狗，即可過關。

**目標**：
- 做出可直接在手機與桌機瀏覽器遊玩的單頁網頁遊戲
- 核心玩法簡單、3 秒內看懂、15 秒內進入第一關
- 以可愛、好分享、容易重玩的短關卡體驗為主

---

## 2. 市場 / 玩法參考整理

我先根據公開資料確認，這類遊戲的主流玩法確實是：

1. **玩家畫一條線**，建立保護牆
2. **蜜蜂從蜂巢飛出**，朝狗狗進攻
3. **線條作為物理碰撞體**，阻擋蜜蜂
4. **撐過固定秒數**（常見為 5–10 秒）即通關
5. 許多版本強調：
   - 單筆畫完成
   - 線條不能畫到狗狗身上
   - 同一關可能有多種解法
   - 後期會加入炸彈、尖刺、熔岩、水等額外危險物

### 參考來源重點

- **CrazyGames / Save My Pets**：描述為「畫線形成牆，保護狗狗不被憤怒蜜蜂攻擊；撐住幾秒即可過關」
- **Apple App Store / Save The Doge**：明確寫到「畫 1 條線保護狗狗」、「蜜蜂攻擊時需保護 10 秒」、「不可畫線穿過狗狗」
- **Google 搜尋結果彙整**：大量同類作品都以「draw line + protect dog from bees」為核心賣點，屬於非常成熟的超休閒玩法

**結論**：
你要做的就是這個成熟品類的 **網頁版變體 / 精簡版**。最適合先做 **MVP：只做狗狗 + 蜜蜂 + 畫線防禦 + 計時過關**，不要一開始就塞太多陷阱類型。

## 5. 核心玩法定義

## 5.1 核心循環（Core Loop）
1. 進入關卡
2. 看見狗狗、蜂巢、地形
3. 玩家按下並拖曳畫線
4. 放開後線條固定
5. 蜜蜂開始或持續攻擊
6. 倒數結束前狗狗沒被碰到 → 過關
7. 若狗狗被蜜蜂碰到 → 失敗重來

## 5.2 核心規則
- 玩家每關只能畫 **1 次**（MVP 建議）
- 線條必須是 **單筆連續**
- 線條長度有限制（增加解謎性）
- 線條不能起始或穿過狗狗碰撞區
- 線條生成後成為固定剛體 / 靜態碰撞體
- 蜜蜂會朝狗狗或保護線附近移動並碰撞反彈
- 玩家只要撐過指定秒數即獲勝

## 5.3 失敗條件
- 蜜蜂碰到狗狗
- 玩家畫線非法（畫到狗狗、超出可畫範圍）
- 玩家未成功形成有效保護，導致倒數結束前失守

## 5.4 勝利條件
- 在倒數時間（建議 6–10 秒）結束前，狗狗未受攻擊

## 7. 關卡設計原則

### 7.1 關卡結構
MVP 建議先做 **20 關**：

- **1–5 關：教學期**
  - 狗狗位置明確
  - 蜂巢單一
  - 玩家學會圈住 / 擋住

- **6–12 關：變化期**
  - 狗狗更靠近牆面或角落
  - 蜂巢位置改變
  - 需要利用地形減少畫線長度

- **13–20 關：挑戰期**
  - 蜜蜂數量增加
  - 開口方向更刁鑽
  - 線長限制更嚴格

### 7.2 關卡原則
- 每關最好能 3–10 秒內看懂
- 至少保留 1 個直覺解法
- 偶爾允許搞笑解法，增加分享感
- 避免解法過度精準，降低挫折

## 8. 美術方向（先定方向，提示詞之後再補）

你已經決定之後用 **ChatGPT image-2** 生成素材，這很適合這種專案。

### 8.1 建議美術風格
- 2D 卡通
- 可愛、明亮、簡潔
- 線條清楚，方便小尺寸辨識
- 角色表情誇張一點，增加喜感

### 8.2 需要的素材清單
- 狗狗待機圖
- 狗狗緊張 / 害怕圖
- 狗狗過關開心圖
- 蜜蜂飛行圖（可 1–2 幀）
- 蜂巢圖
- 地形塊 / 平台圖
- 背景圖
- 成功 / 失敗 UI 面板
- 按鈕（開始、重玩、下一關）

### 8.3 美術製作原則
- 角色與背景分層
- 透明背景 PNG / WebP 為主
- 同一批素材保持視角與光影一致
- 線條不要太細，避免手機上糊掉

## 9. 技術需求

### 9.1 MVP 指定技術棧
**前端語言：**
- TypeScript

**遊戲框架：**
- **Phaser**

**物理引擎：**
- **Matter.js（使用 Phaser 內建整合）**

**資料格式：**
- 關卡資料使用 JSON

**平台目標：**
- 手機瀏覽器優先
- 桌機瀏覽器相容

### 9.2 為什麼選這套
這類遊戲本質上需要：
- 2D 圖像渲染
- 畫線輸入
- 碰撞偵測
- 簡單物理效果
- 手機觸控支援
- 畫面縮放與場景管理

經技術棧查核後，MVP 最適合直接採用 **Phaser + Matter.js**，原因如下：
- Phaser 已具備完整的 scene、input、scale、UI、asset workflow
- Phaser 已提供 Matter 整合，能直接建立 sprite / image / static body / world bounds
- 相比 PixiJS + Matter.js，Phaser 可減少大量框架層整合成本
- 對超休閒網頁遊戲 MVP 來說，更快做出穩定可玩版本

### 9.3 不採用方案（MVP 階段）
- **PixiJS + Matter.js 不作為首版主技術棧**

原因：
- PixiJS 很適合做高效 2D 渲染
- 但 scene、input、UI、遊戲流程管理需要自行補更多結構
- 對目前這個專案的首版目標來說，工程成本不如 Phaser 划算

### 9.4 畫線碰撞體實作策略
這個遊戲最關鍵的技術點不是單純「能不能畫線」，而是 **畫線之後如何穩定轉成可碰撞的保護牆**。

MVP 建議策略：
- 玩家拖曳時先記錄 pointer path
- 對 path 進行 **採樣與簡化**
- 最後轉成 **有限數量的線段碰撞體（segment bodies）**
- 不直接把高密度手勢點位轉成超複雜 polygon body

原因：
- 可降低頂點數與碰撞負擔
- 可減少穿模與奇怪反彈
- 比直接生成複雜 concave body 更適合手機瀏覽器

建議限制：
- 每條線有最大總長度
- 每條線有最大節點數
- 線條厚度固定
- 非法畫線（碰到狗狗 / 超長 / 超出範圍）立即阻止提交

### 9.5 主要系統模組
- Scene / State 管理
- Input 畫線系統
- Line 採樣 / 簡化系統
- Line 轉碰撞體系統
- Bee AI / 移動系統
- Collision 判定
- Win / Lose 狀態管理
- Level data JSON
- UI HUD 系統
- Audio 系統
- Responsive Scale 系統

### 9.6 效能與穩定性要求
- 物理更新以 **固定 timestep（60Hz）** 為優先
- MVP 單關蜜蜂數量先控制在 **5–12 隻**
- 單條線的 segment 數量需設上限
- 低階手機目標至少 **30 FPS**
- 主要目標裝置上互動延遲需低，畫線不得明顯卡頓

### 9.7 畫面尺寸與縮放策略
- 採用 **固定邏輯解析度 + FIT** 模式
- 以直式手機體驗優先設計
- 關卡座標系固定，避免不同裝置尺寸造成關卡解法偏移
- 桌機版採等比顯示並保留邊界留白

## 15. 風險與注意事項

### 15.1 玩法風險
- 若線條太自由，玩家可能大多數關都直接畫大圈解掉
- 若物理太混亂，玩家會覺得結果不公平
- 若解法太刁鑽，超休閒玩家流失很快

### 15.2 技術風險
- 自由畫線轉碰撞體容易出現穿模或效能問題
- 行動裝置上長線段碰撞可能造成卡頓
- 蜜蜂群體 AI 若太複雜，不值得 MVP 成本

### 15.3 對策
- 限制線長
- 多利用地形
- 蜜蜂行為保持簡單但可預測
- 線條可做平滑與節點數限制
